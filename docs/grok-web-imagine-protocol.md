# Grok Web Imagine — Reverse-Engineered Protocol (GW1)

**Captured:** 2026-07-20, live logged-in grok.com/imagine session, via Playwright
`connectOverCDP` + CDP `Network` domain (WebSocket frame + streaming-body capture).
**Purpose:** the exact request/response for image-generate, video-generate, and the
session-health probe, so sudo-ai can replay them server-side on the user's Grok
subscription (GW campaign). All shapes below were observed on real traffic AND
re-verified by headless replay (see §5).

> **Headline correction to the handoff's assumptions:** the generate calls were
> found. **Image generation is NOT a REST call — it rides a WebSocket** and returns
> the image as **inline base64 JPEG frames** (not an `imagine-public.x.ai` URL fetch).
> **Video generation is NOT a `{request_id}`+`search/status` poll — it rides the
> app-chat conversation stream** (`/rest/app-chat/conversations/new`, streaming JSON
> lines) and finishes with a `streamingVideoGenerationResponse` carrying the
> `assets.grok.com` mp4 path.

## 1. Transport / Cloudflare matrix (PROVEN by headless replay)

| Lane | Endpoint | Client that works headless | Cloudflare / auth gate |
|---|---|---|---|
| Health probe | `POST /rest/media/imagine/quota_info` | **curl_cffi** `impersonate="chrome"` + Cookie | plain client → 403; **no** `x-statsig-id` needed |
| **Image gen** | `wss://grok.com/ws/imagine/listen` | **plain `websocket-client`** + Cookie | 101 upgrade succeeds with a valid `cf_clearance`; **no TLS impersonation, no statsig needed** |
| Video step 1 | `POST /rest/media/post/create` | **curl_cffi** `impersonate="chrome"` + Cookie | no `x-statsig-id` needed |
| **Video step 2** | `POST /rest/app-chat/conversations/new` | **curl_cffi** `impersonate="chrome"` + Cookie + **`x-statsig-id`** | **`x-statsig-id` REQUIRED** — without it → **403**; with it → 200 |
| Result assets | `GET https://imagine-public.x.ai/...` (image), `GET https://assets.grok.com/...` (video) | curl_cffi + Cookie | — |

**Two hard, non-obvious facts discovered:**

1. **curl_cffi's WebSocket support is broken in this environment** — `ws.recv()`
   raises `CurlError: WS_RECV, curl (52)` (GOT_NOTHING) on **every** server,
   including `wss://ws.postman-echo.com/raw` (isolated: it is a curl_cffi/libcurl
   issue, not grok-specific). **Use `websocket-client` (`create_connection`) for the
   image WS.** It connected (HTTP 101), received the `session` frame, and streamed
   image blobs with only the Cookie header — Cloudflare does not require Chrome-JA3
   for the WS upgrade once `cf_clearance` is valid.
2. **`x-statsig-id` gates `app-chat/conversations/new` (video) but NOT the REST
   probe/post-create.** `quota_info` carries `x-statsig-id` in the browser yet
   returns 200 when replayed without it; `app-chat` returns 403 without it and 200
   with it. `x-statsig-id` is generated by Grok's obfuscated client JS. **It is
   reusable** — a value captured ~20s earlier, replayed with a fresh
   `x-xai-request-id` on a new conversation, still returned 200 (so: capture once at
   browser spin-up, reuse within a window, re-capture on 403). Length observed: 94.

## 2. Session-health probe (unchanged from handoff, re-verified)

```
POST https://grok.com/rest/media/imagine/quota_info
Headers: Content-Type: application/json, Cookie: <grok.com cookies>,
         User-Agent: <captured UA>, Origin: https://grok.com,
         Referer: https://grok.com/imagine
Body: {}
```
→ `200`:
```json
{"image":{"available":true,"windowSizeSeconds":64800},
 "imagePro":{"available":true,"windowSizeSeconds":64800},
 "imageEdit":{"available":true,"windowSizeSeconds":64800},
 "video":{"available":true,"windowSizeSeconds":64800},
 "video720p":{"available":true,"windowSizeSeconds":64800}}
```
`windowSizeSeconds: 64800` = the 18-hour free quota window. This is the ideal cheap,
side-effect-free health/quota probe.

Wrong path (reached backend, bad route) returns gRPC `{"code":5,"message":"Not Found"}`
(HTTP 404). Distinguish this from a Cloudflare 403 (HTML "Just a moment").

## 3. IMAGE generation — WebSocket `wss://grok.com/ws/imagine/listen`

**Handshake:** standard RFC-6455 upgrade (`Sec-WebSocket-Version: 13`), HTTP/1.1.
Load-bearing headers: `Cookie` (all grok.com cookies, esp. `cf_clearance`, `sso`,
`sso-rw`, `x-userid`), `Origin: https://grok.com`, `User-Agent`. No subprotocol.

**Client → server frames** (JSON text). First a reset, then the prompt:
```json
{"type":"conversation.item.create","timestamp":<ms>,"item":{"type":"message","content":[{"type":"reset"}]}}
```
```json
{"type":"conversation.item.create","timestamp":<ms>,"item":{"type":"message",
 "content":[{"requestId":"<uuid>","text":"<PROMPT>","type":"input_text",
   "properties":{"section_count":0,"is_kids_mode":false,"enable_nsfw":false,
     "skip_upsampler":false,"enable_side_by_side":true,"is_initial":false,
     "aspect_ratio":"9:16","enable_pro":false,"num_generations":8}}]}}
```
- `aspect_ratio`: e.g. `"9:16"`, `"1:1"`, `"16:9"`.
- `num_generations`: images to produce (UI default 8; replay works with 1).
- `enable_pro:true` → the `imagePro` tier; `type:"input_scroll"` (same shape) is the
  infinite-scroll "load more" pagination — not needed for a single generation.

**Server → client frames** (JSON text):
```json
{"type":"session","conversation_id":"<uuid>"}                      // once, on connect
{"type":"json","current_status":"start_stage","percentage_complete":0.0,
 "job_id":"<uuid>","prompt":"...","full_prompt":"...","section_id":0, ...}
{"type":"image","blob":"<base64 JPEG>"}                            // progressive: preview then full-res
{"type":"json","current_status":"completed","percentage_complete":100.0,"job_id":"<uuid>", ...}
```
- Each requested generation gets its own `job_id`. Frames interleave across jobs.
- `image` frames are **progressive**: an early ~32 KB low-res preview
  (`/9j/4AAQ...` = JFIF), later a full-res ~200–400 KB blob (`/9j/4QJA...` = EXIF).
  Keep the largest blob per `job_id`, or take the blob that arrives with/after that
  job's `completed` status.
- The final image is ALSO retrievable at
  `https://imagine-public.x.ai/imagine-public/images/<job_id>.jpg` (this URL is what
  the video step consumes).

**Replay verified:** `websocket-client` + Cookie produced valid JPEGs headless
(`ffd8ffe1`, 136 KB for a 1:1 single generation). See §5.

## 4. VIDEO generation — REST kickoff + app-chat stream

Triggered in the UI by the per-image **"Make video"** button. Three calls:

**4.1** `POST /rest/media/post/create` (curl_cffi, Cookie) — publish the source image
as a post so it has an id:
```json
{"mediaType":"MEDIA_POST_TYPE_IMAGE",
 "mediaUrl":"https://imagine-public.x.ai/imagine-public/images/<image_job_id>.jpg"}
```
→ `200 {"post":{"id":"<post_id>", "mediaUrl":"...", "resolution":{"width":720,"height":1280}, ...}}`
(`post_id` == the image `job_id`).

**4.2** `POST /rest/app-chat/conversations/new` (curl_cffi, Cookie, **`x-statsig-id`
REQUIRED**, streaming response):
```json
{"temporary":true,"modelName":"imagine-video-gen",
 "message":"https://imagine-public.x.ai/imagine-public/images/<image_job_id>.jpg  --mode=normal",
 "enableSideBySide":true,
 "responseMetadata":{"experiments":[],"modelConfigOverride":{"modelMap":{
   "videoGenModelConfig":{"parentPostId":"<post_id>","aspectRatio":"9:16",
     "videoLength":6,"resolutionName":"720p"}}}}}
```
(note the two spaces before `--mode=normal`; `videoLength` in seconds; `resolutionName`
`"720p"`.)

**4.3** Response = streaming JSON lines (chunked HTTP, one JSON object per line):
```json
{"result":{"conversation":{"conversationId":"...","temporary":true, ...}}}
{"result":{"response":{"userResponse":{ ... "mediaGenInput":{"imageToVideo":{"prompt":"<img url>","inputAssets":["<post_id>"],"aspectRatio":"9:16","duration":6,"resolutionName":"720p","modelName":"imagine-video-gen","mode":"normal"}} ...}}}
{"result":{"response":{"progressReport":{"category":"PROGRESS_REPORT_CATEGORY_ATTACHMENTS_PREPROCESSING","state":"PROGRESS_REPORT_STATUS_PENDING","message":"Started pre-processing attachments"} ...}}}
{"result":{"response":{"streamingVideoGenerationResponse":{"videoId":"<video_id>","progress":0,"parentPostId":"<post_id>","imageReference":"https://assets.grok.com/users/<uid>/generated/<image_job_id>/image.jpg","height":1280,"width":720,"resolutionName":"720p"} ...}}}
... progress 5,16,33,50,66,94,95 ...
{"result":{"response":{"streamingVideoGenerationResponse":{"videoId":"<video_id>","progress":100,"assetId":"<video_id>",
   "videoUrl":"users/<uid>/generated/<video_id>/generated_video.mp4",
   "thumbnailImageUrl":"users/<uid>/generated/<video_id>/preview_image.jpg", ...}}}}
{"result":{"response":{"modelResponse":{"message":"I generated a video","sender":"ASSISTANT","fileAttachments":["<video_id>"], ...}}}}
```
**Final video URL:** `https://assets.grok.com/` + `videoUrl`, i.e.
`https://assets.grok.com/users/<uid>/generated/<video_id>/generated_video.mp4`
(`mp4`, `ftyp` magic; ~6 s clip). Thumbnail analogously under `preview_image.jpg`.
Poll the stream to `progress:100`; ~25–45 s wall time observed.

> `POST /rest/media/search/status {}` seen alongside is **media-search indexing**
> (`MEDIA_SEARCH_INDEX_STATUS_NOT_ELIGIBLE`), NOT the video-generation poll — do not
> use it to detect video completion. Completion comes from the app-chat stream.

## 5. Headless replay evidence (2026-07-20, this host, same IP as the browser)

- **Probe:** `curl_cffi` `impersonate="chrome"` + captured Cookie →
  `200` quota JSON (all tiers `available:true`).
- **Image:** `websocket-client` `create_connection` + Cookie → HTTP 101, `session`
  frame, `start_stage`→`completed` status frames, base64 `image` blobs; largest blob
  decoded to a valid JPEG (`ffd8ffe1`, saved `/tmp/gw-proof-image.jpg`, 136 037 B).
  **No TLS impersonation and no `x-statsig-id` used.**
- **Video:** `curl_cffi` `POST /rest/media/post/create` → 200 (`post_id`);
  `POST /rest/app-chat/conversations/new` with a **fresh `x-statsig-id`** → 200
  streaming (without statsig → 403; statsig reused ~20 s later on a new conversation
  → still 200).

## 6. Cookies to capture (grok.com domain)

Load-bearing: `sso`, `sso-rw` (long-lived login, weeks), `cf_clearance` (Cloudflare
clearance, hours, IP-bound), `__cf_bm` (~30 min), `x-userid`, `grok_device_id`,
`team-id`. Carry **all** grok.com cookies verbatim to be safe. Capture the exact
`User-Agent` (must match on replay). Plus **capture a fresh `x-statsig-id`** at
spin-up for the video lane (reusable within a window; re-capture on app-chat 403).

## 7. Error taxonomy (for the replay bridge)

- **403 + HTML "Just a moment"** → Cloudflare block → refresh `cf_clearance`/`__cf_bm`
  (headless spin-up).
- **403 on `app-chat` with valid cookies** → missing/stale `x-statsig-id` → re-capture
  statsig (headless spin-up), retry.
- **404 + `{"code":5,"message":"Not Found"}`** → wrong path (reached backend) → fix
  the route, do NOT refresh cookies.
- **401 / login HTML** → `sso` dead → `needsRelogin`.
- **WS 101 then immediate close / no frames** → cookie (esp. `cf_clearance`) stale, or
  (spuriously) curl_cffi's broken WS — use `websocket-client`.
